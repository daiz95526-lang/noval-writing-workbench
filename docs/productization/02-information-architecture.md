# NOVAL Phase 1 信息架构

状态：待用户确认  
依赖：[01-product-definition.md](01-product-definition.md)

## 1. 设计目标

信息架构必须解决四个问题：

1. 用户进入产品后知道从哪里开始。
2. 用户始终知道当前项目、当前章节和当前创作阶段。
3. 分析、规划、生成、编辑、质检和正式保存形成连续流程。
4. 普通用户看不到任务 ID、JSON、内部路径和模型底层参数。

设计不以“让每个后端能力都有菜单”为目标，而以用户任务为中心组织入口。

## 2. 层级模型

NOVAL 采用两层信息架构。

### 2.1 全局层

不依赖当前项目，负责：

- 开始或继续工作；
- 创建、选择、归档和恢复项目；
- 配置默认模型、数据与隐私；
- 查看全局任务和环境状态。

全局层主入口：

1. 首页
2. 项目
3. 设置

### 2.2 项目层

选择项目后出现，所有内容必须明确绑定当前项目：

1. 项目概览
2. 语料库
3. 分析
4. 创作
5. 章节库

项目层不再重复“项目”主入口。页面顶部固定显示项目名称和项目切换器；高级用户可以展开查看 `project_id` 和数据位置。

## 3. 推荐主导航

### 3.1 导航结构

```text
NOVAL
├─ 首页                         全局
├─ 项目                         全局
├─ 当前项目：{项目名称}          项目上下文/切换器
│  ├─ 项目概览
│  ├─ 语料库
│  ├─ 分析
│  ├─ 创作
│  └─ 章节库
└─ 设置                         全局工具
```

这不是八个并列产品模块。首页、项目和设置属于全局层；中间五项只在选中项目后出现，并共享同一项目上下文。

### 3.2 每个主入口的职责

| 入口 | 层级 | 职责 | 不承担的职责 |
| --- | --- | --- | --- |
| 首页 | 全局 | 最近项目、进行中任务、首次引导、全局异常和继续工作 | 不展示空白语料统计，不编辑项目内容 |
| 项目 | 全局 | 创建、搜索、打开、归档、备份和恢复项目 | 不直接生成章节 |
| 项目概览 | 项目 | 项目准备度、创作进度、最近章节、下一步建议 | 不堆放所有编辑表单 |
| 语料库 | 项目 | 导入、扫描、校验和浏览项目语料 | 不承担知识库编辑和生成 |
| 分析 | 项目 | 风格、人物、地点、设定、主题和情节知识 | 不直接保存正式章节 |
| 创作 | 项目 | 总体构想、章节规划、章节写作工作流 | 不展示全项目历史管理 |
| 章节库 | 项目 | 候选稿、正式章节、版本和导出 | 不负责模型配置或语料扫描 |
| 设置 | 全局/项目覆盖 | 模型、数据、隐私、备份、诊断和高级参数 | 不作为日常创作入口 |

## 5. 创作区内部结构

“创作”是一个流程容器，不是把所有功能放在一页。

### 5.1 总体构想

- 输入创作方向或使用已有知识；
- 生成总体构想；
- 用户编辑、重新生成、比较和确认；
- 只有确认后，才标记为可生成完整章节规划。

### 5.2 章节规划

- 展示全书章节及状态；
- 支持自动生成完整规划和人工编辑；
- 用户选择当前要写的章节；
- 显示上一章/下一章和连续性依赖。

### 5.3 章节工作区

章节生成、编辑、修改和质检共享同一个“当前章节”上下文：

```text
计划摘要 → 生成候选稿 → 人工编辑 → 修改候选 → 质检 → 正式保存
```

它们可以表现为步骤、面板或子路由，但不应成为四个一级导航项。页面切换不能丢失当前文本。

## 6. 当前页面到新页面的映射

| 当前页面/区域 | 新位置 | 处理方式 |
| --- | --- | --- |
| 系统总览 | 首页 + 项目概览 | 拆分全局最近项目与单项目统计 |
| 章节列表（总览） | 语料库/章节列表 | 与语料来源放在一起 |
| 语料管理 | 语料库 | 重命名并去除固定路径/文件数文案 |
| 风格分析 | 分析/风格分析 | 作为分析中心子页面 |
| 知识库构建 | 分析/知识库 | 从生成页面移出 |
| 续写工作台：总体构想 | 创作/总体构想 | 改为通用作品构想 |
| 续写工作台：章节生成 | 创作/章节规划 + 章节工作区 | 规划选择与正文工作分离 |
| 续写工作台：临时记录 | 章节库/候选稿 | 与草稿、生成结果统一语义 |
| 续写工作台：正式章节 | 章节库/正式章节 | 正式内容独立管理 |
| 章节完整性检查 | 章节工作区/审核 | 生成后自动执行并可手工重跑 |
| AI 深度质检 | 章节工作区/AI 质检 | 当前章节审核阶段 |
| 定向修改/整章重写 | 章节工作区/修改 | 产生新候选版本，不直接覆盖 |
| 草稿版本 | 章节编辑器/版本历史 | 当前章节上下文内管理 |
| 导出 | 章节详情、章节库、项目菜单 | 上下文操作，不设一级页面 |
| 配置状态 | 设置/模型与连接 | 用可理解的检查结果替代技术字段 |
| 任务日志 | 全局任务抽屉 + 任务详情 | 默认展示阶段，技术日志折叠 |

## 7. 功能入口规则

### 7.1 不应放在主导航

- 版本历史：属于当前章节或项目备份。
- 导出：属于章节、章节库或项目操作。
- AI 质检：属于当前章节工作流。
- 临时任务日志：属于任务详情或诊断。
- 配置状态：属于设置中的连接检查。
- JSON 查看、重解析、缓存清理：属于高级设置。
- 上传单文件、删除章节：属于语料库上下文操作。

### 7.2 必须放在项目上下文

- 语料、章节、分析、知识库；
- 总体构想和章节规划；
- 候选稿、正式章节和版本；
- 项目级模型覆盖；
- 项目任务和项目备份。

任何读取或写入这些对象的 API 和页面都必须有明确 `project_id`。前端不得仅依赖“最后打开项目”隐式决定写入目标。

### 7.3 全局工具

- 默认模型和连接；
- 数据根目录；
- 全局任务汇总；
- 备份与恢复入口；
- 应用版本、日志和环境诊断。

## 8. 项目上下文规范

### 8.1 固定可见信息

项目页面顶部始终显示：

- 项目名称；
- 项目类型：续写/原创/分析；
- 当前准备状态；
- 项目切换入口。

章节工作区额外显示：

- 当前章节序号与标题；
- 当前状态：未开始/候选稿/审核中/已正式；
- 上一章和下一章；
- 自动保存状态。

### 8.2 面包屑

推荐格式：

```text
项目名称 / 创作 / 第 12 章 / AI 质检
```

不要显示：

```text
writing_projects/longzu6/official_chapters/chapter_012.json
```

实际路径仅在“查看数据位置”中提供。

### 8.3 项目切换保护

- 有未保存编辑时，切换项目必须先自动保存草稿或明确提示。
- 有进行中任务时，切换项目不取消任务；任务继续运行并显示所属项目。
- 返回项目时恢复当前章节、文本、任务和滚动位置。
- 项目切换器不显示内部 ID 作为主标签。

## 9. 状态与反馈架构

### 9.1 项目准备状态

```text
新建
→ 模型待配置
→ 语料待导入（原创项目可跳过）
→ 待分析（可跳过但提示）
→ 可规划
→ 创作中
→ 已归档
```

### 9.2 章节状态

```text
未规划
→ 已规划
→ 生成中
→ 候选稿
→ 修改中
→ 待质检
→ 待确认
→ 正式章节
```

失败、取消和部分成功是任务状态，不直接替代章节状态。任务失败时，章节仍保留上一个稳定状态和部分候选稿。

### 9.3 反馈层级

| 层级 | 用途 | 示例 |
| --- | --- | --- |
| 页面状态 | 当前页面无法继续 | 模型未配置、项目不可读 |
| 任务状态 | 长任务进度和恢复 | 正在分析第 12/30 章 |
| 操作反馈 | 短操作结果 | 已保存版本 |
| 字段错误 | 输入校验 | 项目名称不能为空 |
| 高级诊断 | 技术排查 | Provider 返回 429 |

## 10. URL 与页面状态建议

Phase 1 只定义语义，不实施路由。建议 Phase 2/后续采用显式项目 URL：

```text
/
/projects
/projects/new
/projects/{project_id}/overview
/projects/{project_id}/corpus
/projects/{project_id}/analysis
/projects/{project_id}/create/idea
/projects/{project_id}/create/plan
/projects/{project_id}/chapters/{chapter_id}/workspace
/projects/{project_id}/library/drafts
/projects/{project_id}/library/official
/settings/models
/settings/data
/settings/advanced
```

URL 中使用稳定 `project_id`，显示名称可修改。章节编辑内容和运行中任务不能只存在于 React 组件内存。

## 11. 首次使用的信息架构

第一次启动不进入空白统计页，而进入可恢复向导：

```text
欢迎
→ 创建项目
→ 选择项目类型
→ 配置模型（可稍后，但生成前必须完成）
→ 导入语料 / 使用原创示例 / 创建空项目
→ 扫描结果
→ 推荐下一步
```

向导完成后进入项目概览。高级路径、端口、token 和日志折叠在“高级设置”，不阻塞普通用户。

## 12. 信息架构原则检查

| 原则 | 设计响应 |
| --- | --- |
| 始终知道当前项目 | 固定项目标题、切换器和显式项目 URL |
| 始终知道当前章节 | 章节工作区固定章节上下文 |
| 知道流程阶段 | 项目准备状态、章节状态和下一步动作 |
| 主要按钮明确 | 每页只设置一个主行动 |
| 高级参数隐藏 | 统一进入高级设置与诊断 |
| 临时/正式严格区分 | 候选稿与正式章节分区、确认后转换 |
| 生成/修改/质检/保存清晰 | 单章节工作流顺序化 |
| 不理解 task ID | 展示任务名称、阶段和所属项目 |
| 不理解 JSON | 默认使用表单、卡片和可读报告 |
| 不理解目录 | 路径仅作可选信息 |
| 页面不过载 | 总体构想、规划、章节工作区、章节库分责 |

## 13. Phase 2 架构输入

信息架构要求 Phase 2 至少解决：

1. 所有项目资源有稳定 `project_id`。
2. API 使用显式项目上下文，避免全局单例隐式写入。
3. 前端有可恢复的当前项目和当前章节状态。
4. 任务记录包含项目、资源类型和资源 ID。
5. 候选稿与正式章节拥有明确状态和转换规则。
6. 旧 `longzu` 数据可作为一个项目读取，且默认不移动、不删除。
7. 项目路径经过规范化和路径穿越防护。
8. 项目 manifest 有 schema version、创建时间、更新时间和迁移状态。

实现前必须先确认 [01-product-definition.md](01-product-definition.md) 第 8 节列出的决策。
