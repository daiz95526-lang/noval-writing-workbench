# NOVAL Phase 2 工程实施与验收记录

状态：实现完成，等待最终自动化和人工验收

## 1. 范围

Phase 2 只建立通用项目模型、项目目录、兼容迁移、项目级 API/任务/缓存和最低限度前端项目上下文。不进行视觉重构，不删除旧数据，不进入 Phase 3。

## 2. 硬编码审计与处理

| 位置 | 旧用途 | Phase 2 处理 | 兼容方式 |
| --- | --- | --- | --- |
| `config.py` | `longzu6`、旧语料和写作目录默认值 | 保留 | 仅作为现有安装默认值，可由 `.env` 覆盖 |
| `writing_project_store.py` | 旧 manifest、Book Plan 文件名和路径标签 | 保留并按 `legacy_layout` 分支 | 托管项目使用通用文件名和项目目录 |

## 3. 后端实现

- `Project` schema、稳定 ID、项目状态、语料配置和迁移状态。
- `ProjectStore` 管理项目清单、目录、归档、备份、软删除、迁移和摘要。
- `safe_child` 防止托管路径穿越；文件对象 ID 使用白名单。
- `EXTERNAL_CORPUS_ROOTS` 控制外部只读语料授权，manifest 读取时再次校验。
- `ProjectScopedStore` 和 `ProjectScopedDict` 隔离存储与内存缓存。
- FastAPI 中间件统一建立项目上下文，后台任务显式恢复项目。
- 原子写入使用同目录唯一临时文件后 `replace`。
- 迁移失败目标自动移出活动项目根，旧数据保持原位。

## 4. 前端最低限度支持

- 启动时读取项目列表并验证当前项目。
- 支持创建、选择和切换项目。
- 普通 API 和导出请求携带 `X-Project-ID`。
- 切换项目通过 key 重新挂载业务页面。
- 风格分析和创作工作台的 `localStorage` 键包含 `project_id`。
- 项目列表验证完成前不挂载业务页，避免错误项目的预加载请求。

本阶段没有调整页面布局、导航视觉或组件样式。

## 5. API 兼容策略

项目管理使用 `/api/projects`。现有语料、分析、生成、草稿、规划、章节和导出路由保持 URL 不变，通过 `X-Project-ID` 明确项目。无请求头时兼容旧项目；不存在或非法项目返回清晰的 404/400。

## 6. 安全约束

- 不接受非法或穿越形式的项目 ID。
- 托管路径必须位于 `PROJECTS_DIR`。
- 外部语料必须在明确允许根下，且保持只读配置。
- 项目删除要求二次确认并先备份；旧项目禁止 API 删除。
- 异常文本中的 API Key、Authorization 和 Bearer 值会脱敏。
- 项目 manifest 不保存真实 API Key。
- 不支持多个后端进程同时写同一个数据根。

## 7. 测试策略

后端测试必须使用 `tmp_path` 或临时目录，覆盖：

- 两项目语料、任务和正式章节隔离；
- 标题修改不改变 ID 和路径；
- 非法 ID、`..` 和未授权绝对路径被拒绝；
- 显式允许的外部目录可用；
- 通用项目不使用旧作品 prompt、卷映射或规则角色；
- legacy 项目保持旧行为；
- 成功迁移统计一致且语料不变；
- 迁移统计失败自动回退；
- 删除一个测试项目不影响另一个项目。

前端执行 ESLint、TypeScript `--noEmit` 和生产构建。完整验收还执行 Ruff 与 `git diff --check`，并比较受保护目录执行前后的文件数、总大小和最新修改时间。

## 8. 已知限制与后续边界

- 任务不跨后端重启持久化。
- 没有跨进程文件锁，部署时只能运行一个写入实例。
- 前端只有最低限度项目选择，没有实施 Phase 1 的完整信息架构和视觉方案。
- 外部目录授权目前通过 `.env`，没有文件选择器。
- 旧兼容模板仍保留特定作品文本，但托管项目运行时测试确保不会使用这些内容。

以上限制不在 Phase 2 内继续扩展功能；视觉、完整导航和产品页面改造属于后续阶段，必须另行确认。
