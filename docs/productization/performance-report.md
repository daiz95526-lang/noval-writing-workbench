# NOVAL Phase 5 性能报告

状态：实现完成，等待人工验收

## 1. 优化内容

- 新增 `/api/corpus/chapters/page`，支持卷、关键字、offset 和 limit。
- 任务列表支持项目隔离、状态过滤、offset 和 limit。
- 新增临时记录与正式章节分页摘要接口，列表响应不包含完整正文和大型生成参数。
- 创作工作台先加载最多 200 条摘要，点击记录后再按 ID 加载正文。
- 语料列表仍只返回 `ChapterMeta`，正文由章节详情接口按需加载。
- 已有项目级语料运行时缓存保留；只有显式扫描或项目切换后才重新加载。
- 日志、任务历史和文件备份都有保留上限，避免无限增长。

## 2. 合成基准

环境：本地 Windows、Python 3.12；数据全部为内存生成的原创测试字符串，不读取用户语料。

| 场景 | 结果 |
| --- | ---: |
| 200 条记录，每条 8,000 字，完整 JSON | 2,348,070 bytes |
| 同一批记录的摘要 JSON | 97,070 bytes |
| 摘要载荷减少 | 95.87% |
| 完整 JSON 序列化 | 8.320 ms |
| 摘要 JSON 序列化 | 0.368 ms |
| 350 章元数据取前 100 章 | 24,350 bytes / 0.104 ms |

该结果只比较序列化和载荷，不代表端到端网络延迟；实际收益还受磁盘、浏览器和正文长度影响。

## 3. 长文本与大项目

- 8,000+ 字正文仍通过单独详情接口加载，避免与全部历史记录一起传输。
- 编辑器继续使用受控文本区和项目级本地草稿恢复，没有引入富文本编辑器或新运行时依赖。
- 333+ 章节通过元数据分页接口处理；现有选择器保留全量轻量元数据以支持跨章上下文选择。
- 搜索在分页章节接口服务端执行，避免前端先下载正文。

## 4. 已知限制

- 创作工作台当前显示最近 200 条历史记录，分页 API 可读取更早记录，但 UI 尚无“加载更多”入口。
- 没有引入虚拟列表。当前摘要列表规模下收益不足以抵消复杂度；超过数千条记录后应重新评估。
- 没有启用响应压缩中间件。正文按需加载后优先级下降，部署在非 loopback 网络时再评估。
- JSON 文件存储不适合百万级记录；当前上限和本地单用户场景不需要 SQLite 索引。

## 5. 发布阻塞

没有发现 333 章或 8,000 字章节的结构性阻塞。人工验收应重点检查拥有数百条生成历史时的首屏时间和浏览器内存。
