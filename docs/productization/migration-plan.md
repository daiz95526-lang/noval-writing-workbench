# NOVAL Phase 2 旧项目迁移与回滚方案

状态：迁移能力已实现；未对真实本地项目执行迁移

## 1. 原则

1. 旧目录继续原地可读，不自动迁移。
2. 不删除、不移动、不改写原始 `source_txt`。
3. 不删除旧正式章节、临时记录、版本、规划或导出。
4. 用户必须先查看预览，再提交包含旧项目 ID 的明确确认。
5. 默认引用原始语料，避免重复复制大文件。
6. 任何失败都不能让未完成目标出现在活动项目列表。

## 2. 兼容识别

配置中的 `PROJECT_ID`、`CORPUS_SOURCE_DIR`、`CONTINUATION_PROJECT_DIR` 和 `WRITING_PROJECT_DIR` 共同描述旧项目。只要旧目录之一存在，项目列表就提供一个 `storage_mode=legacy` 的“本地续写项目”。

兼容读取仍使用原路径和旧文件名，包括旧 Book Plan 文件名。通用项目代码不把旧作品名称作为产品标题。

## 3. 迁移预览

`GET /api/projects/{project_id}/migration-preview` 只读取目录元数据和章节索引，返回：

- 章节数和索引总字数；
- 派生文件、可写文件和估算复制大小；
- 原始语料大小；
- 是否复制原始语料；
- 不自动迁移和不删除旧目录的警告。

预览不输出语料正文、草稿正文或 API Key。

## 4. 执行顺序

1. 校验 `confirm_source_project_id` 与旧项目完全一致。
2. 生成迁移预览。
3. 在 `PROJECTS_DIR/.migration_backups` 创建迁移前 ZIP。
4. 创建目标项目并标记 `migration_state=in_progress`。
5. 复制派生数据、分析、规划、草稿、正式章节、版本和导出。
6. `corpus_mode=reference` 时登记只读外部语料引用；`copy` 时才复制语料。
7. 规范目标 manifest 和旧元数据中的项目 ID、标题与通用文件名。
8. 比较迁移前后的章节数和索引总字数。
9. 只有统计一致时才原子写入 `migration_state=complete`。

## 5. 备份范围

迁移前 ZIP 包含章节索引、分析、风格缓存、规划和写作目录。默认不包含原始语料，原因是原始目录保持原地且可能很大。ZIP 使用临时唯一文件名，写完后原子替换为正式备份。

## 6. 失败与回滚

复制、规范化、解析或最终统计任一步失败时：

- 目标 manifest 标记为 `failed`；
- 目标目录移动到 `PROJECTS_DIR/.failed_migrations`；
- 目标项目不再能通过项目 API 打开；
- 迁移前 ZIP 保留；
- 旧项目和旧目录继续可读；
- 原始语料保持不变。

由于迁移只复制或引用旧数据，自动回滚的核心是撤销目标项目注册，而不是反向覆盖旧目录。失败目录只用于诊断，不能自动重新加入项目列表。

## 7. 删除与恢复

迁移成功不会删除旧目录。托管项目删除需要确认值与 ID 完全一致，系统先创建 ZIP，再移动到 `.trash`。本阶段没有永久删除 API；恢复可通过把已验证目录移回项目根完成，正式恢复工具属于后续独立阶段。

## 8. 验证

自动化测试覆盖：

- 成功迁移前后章节数和字数一致；
- 迁移前备份存在；
- 原始语料哈希和时间戳不变；
- 旧项目迁移后继续可读；
- 人为制造统计不一致时，目标移入失败区且不出现在活动项目中；
- 测试只使用 pytest 临时目录。

## 9. 真实数据状态

Phase 2 没有调用真实项目迁移接口。真实旧项目仅执行兼容读取、summary 和迁移预览验证，不写入旧目录。
