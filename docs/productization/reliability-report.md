# NOVAL Phase 5 可靠性报告

状态：实现完成，等待人工验收  
范围：任务、文件写入、错误恢复、模型调用和本地单进程部署

## 1. 任务可靠性

- 任务记录持久化到 `DATA_DIR/tasks/{project_id}/tasks.json`，不写入语料或正式章节目录。
- 每条任务记录包含 `project_id`、操作类型、目标、阶段、进度、部分结果、超时、尝试次数和重试来源。
- 应用重新加载任务文件时，遗留的 `pending` / `running` 自动转为 `interrupted`，不会继续显示为运行中。
- 任务按项目延迟加载，查询、取消和重试都校验 `project_id`。
- 失败、中断和取消任务可通过 `POST /api/tasks/{task_id}/retry` 重试；重试创建新任务并保留 `retry_of`。
- 超过 `TASK_TIMEOUT_SECONDS` 的任务转为失败，错误码为 `TASK_TIMEOUT`。
- 完整章节生成的 `partial_text`、分段进度和可接受状态随任务记录同步落盘。

任务文件使用 JSON 是有意选择：当前产品是本地单进程应用，JSON 易审计、备份和恢复，不需要引入数据库迁移。任务历史超过默认 500 条时只保留最新记录。

## 2. 文件可靠性

`backend/app/services/file_ops.py` 统一提供：

1. 同目录临时文件写入；
2. `flush` 和 `fsync`；
3. 写前备份；
4. `os.replace` 原子替换；
5. 每路径进程内锁；
6. 有限备份保留；
7. JSON 损坏时读取最近有效备份并恢复；
8. 项目内软删除。

项目清单、manifest、总体构想、章节规划、草稿、临时记录、正式章节元数据、版本和任务记录均通过统一原子写入函数保存。正式章节覆盖仍会先创建版本快照。

## 3. 错误恢复

- 所有 API 错误返回统一错误码、用户消息、详情、`request_id` 和 `retryable`。
- 应用内部异常不把 Python 堆栈返回前端。
- 任务错误保留类型和可重试属性，不保存 Authorization 或 API Key。
- 前端识别 404、422、408/504 和通用 HTTP 错误；请求超时会结束 loading。
- 文件写入失败时临时文件被清理，旧文件保持不变。

## 4. 已知限制

- 任务文件锁只覆盖当前后端进程。NOVAL 当前只支持一个后端进程，不应使用多 worker 启动。
- 取消为协作式取消。已进入第三方 SDK 同步线程的单次调用可能等待到超时后才释放，但任务状态会立即标记为取消。
- 重试参数和部分正文以本地明文保存。它们受本机文件权限和 `.gitignore` 保护，不提供磁盘加密。
- 备份恢复已有 API 和底层能力，但尚无面向普通用户的恢复页面。
- 回收站没有自动清理周期，长期使用时需由后续维护工具提供容量管理。

## 5. 发布阻塞

没有发现会损坏原文件或导致任务永久 loading 的已知阻塞。正式发布前仍需在目标 Windows 环境人工验证进程强制终止后的任务中断显示和软删除恢复操作。
