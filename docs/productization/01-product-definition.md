# NOVAL Phase 1 产品定义

状态：待用户确认  
版本：Phase 1 draft  
基线：[00-baseline-audit.md](00-baseline-audit.md)

## 1. 产品定位

NOVAL 是：

> 本地优先的 AI 长篇小说分析、规划、续写和章节管理工作台。

它帮助用户把合法取得的本地语料、自己的创作目标和模型能力组织成一条可追踪、可修改、可质检、可恢复的长篇写作流程。NOVAL 的核心不是“替用户写一段文字”，而是长期管理一个小说项目从语料、分析、构想到章节交付的全过程。

### 1.1 核心价值

1. **本地优先**：项目、语料、草稿、正式章节和版本由用户掌控；远程模型调用必须透明且可配置。
2. **长篇一致性**：风格、人物、设定、情节、章节规划和已完成正文共同构成生成上下文。
3. **人机协作**：AI 负责分析、建议和草稿生成，用户负责目标、审核、修改和正式确认。
4. **过程可恢复**：长任务有阶段、进度、错误、重试和保留部分结果；失败不应丢失用户输入。
5. **临时与正式分离**：AI 生成内容默认是候选稿，只有用户确认后才进入正式章节库。
6. **项目隔离**：每部作品是独立项目；现有 `longzu` / `longzu6` 只能作为一个本地项目实例。

### 1.2 产品承诺

- 用户始终知道当前项目、当前章节和当前流程阶段。
- 用户不需要理解 API、JSON、端口、任务 ID 或内部目录才能完成主流程。
- 原始语料保持只读语义；分析和生成结果写入项目自己的工作目录。
- 所有覆盖、删除和正式保存操作都有明确结果和恢复边界。
- 未配置模型或环境异常时，产品说明“缺什么、在哪里修复、修复后如何继续”。

### 1.3 非目标

MVP 不把 NOVAL 定义为：

- 在线多人协作文档平台；
- 小说版权内容分发平台；
- 内置特定作品语料的续写产品；
- 无需用户审核的全自动出版系统；
- 通用聊天机器人或提示词调试器；
- 云端账号、支付、社区和作品市场。

## 2. 核心对象与产品术语

| 术语 | 产品含义 | 用户可见 |
| --- | --- | --- |
| 项目 | 一部长篇作品及其语料、分析、规划、章节和配置的隔离容器 | 是 |
| 语料 | 用户有权使用的参考文本或已有作品文本，默认只读 | 是 |
| 分析 | 对语料的章节结构、风格和知识要素进行提取 | 是 |
| 知识库 | 人物、地点、设定、主题、情节和风格规则的结构化结果 | 是 |
| 总体构想 | 全书方向、核心冲突、主要角色弧线、卷/阶段结构 | 是 |
| 章节规划 | 每章目标、情节节点、角色、连续性要求和状态 | 是 |
| 候选稿 | AI 生成、导入或修改后尚未正式确认的章节内容 | 是 |
| 正式章节 | 用户确认纳入作品主线的章节版本 | 是 |
| 版本 | 对草稿或正式章节的不可变快照 | 是 |
| 任务 | 分析、构建、生成或质检的后台执行记录 | 仅以“进行中的工作”呈现 |
| `project_id` | 项目的稳定内部标识 | 默认隐藏 |
| 文件路径 | 本地数据位置 | 可查看，但不是主操作入口 |

## 3. 目标用户

### 3.1 基于已有作品进行长篇续写的用户

- **目标**：理解原作风格、人物和未完成线索，规划并连续写出后续章节。
- **主要痛点**：原作信息量大；角色声音和世界规则容易漂移；章节之间难以保持连续；AI 常生成短、散、无后续承接的文本。
- **使用频率**：高频，集中创作期可能每天使用。
- **最重要功能**：语料导入、风格分析、知识库、总体构想、章节规划、完整章节生成、连续性检查、正式章节库。
- **不应暴露**：语料内部索引文件、Prompt、token 限制、任务 ID、JSON、后端路径。

### 3.2 分析本地小说语料的用户

- **目标**：查看章节结构、语言风格、人物、地点、主题和情节关系，不一定继续创作。
- **主要痛点**：长文本难以整体分析；手工整理人物和情节成本高；分析结果难以回溯到来源。
- **使用频率**：中低频，通常按作品或分析批次使用。
- **最重要功能**：语料库、章节浏览、风格分析、知识库、分析结果导出或复用。
- **不应暴露**：模型原始响应、缓存目录、解析器错误堆栈、底层 API 路由。

### 3.3 规划原创长篇小说的用户

- **目标**：从原创设定或少量参考资料出发，形成总体构想、章节规划并逐章创作。
- **主要痛点**：前期结构不完整；中后期失去方向；设定、伏笔和角色弧线难以维护。
- **使用频率**：高频，覆盖数周到数月。
- **最重要功能**：创建空项目、手工知识库/设定、总体构想、章节规划、章节编辑、版本和导出。
- **不应暴露**：产品把“导入已有小说”当成强制步骤，也不应显示特定作品名称或固定卷数。

### 3.4 已有草稿、需要 AI 修改和质检的用户

- **目标**：导入或打开自己的草稿，继续写作、定向修改、检查完整性并保留版本。
- **主要痛点**：修改可能破坏原意；AI 重写覆盖原稿；质检结果与正文脱节；缺少修改前后的版本保护。
- **使用频率**：高频，通常围绕单章反复使用。
- **最重要功能**：章节编辑、局部修改、整章重写、AI 深度质检、差异确认、版本快照、正式保存。
- **不应暴露**：模型响应解析细节、内部 generation ID、保存文件名、失败堆栈。

### 3.5 具备技术能力的高级用户

- **目标**：选择模型服务、调整性能参数、自定义数据目录、查看诊断并集成本地工作流。
- **主要痛点**：默认设置不透明；模型兼容差异难定位；数据和日志位置不清晰。
- **使用频率**：配置低频、诊断按需、创作高频。
- **最重要功能**：模型配置、目录设置、连接测试、任务诊断、日志、备份/恢复、导出。
- **可选择暴露**：Provider、模型名、超时、token 上限、实际路径、结构化诊断；仍不显示完整 API Key。

### 3.6 不理解 API、JSON、端口和文件系统的普通用户

- **目标**：按清晰步骤完成创建项目、导入、分析、规划、写作和导出。
- **主要痛点**：不知道从哪里开始；错误信息看不懂；担心正文丢失；不清楚内容保存在哪里。
- **使用频率**：从首次尝试到持续写作，频率差异较大。
- **最重要功能**：首次引导、默认配置、清晰主按钮、自动保存、进度、可恢复错误、最近项目和下一步建议。
- **不应暴露**：API Key 值、端口、环境变量名、JSON、任务 ID、内部目录、堆栈、模型原始响应。

## 4. 核心使用场景

### 场景 A：第一次使用

启动 NOVAL → 欢迎与隐私说明 → 创建项目 → 配置模型并测试连接 → 选择“导入语料”或“原创空项目” → 扫描章节 → 查看导入结果 → 开始分析或进入创作。

成功标准：用户在不阅读开发文档的情况下创建第一个可继续工作的项目，并知道下一步是什么。

### 场景 B：续写现有作品

选择项目 → 查看项目准备状态 → 检查语料和分析结果 → 生成总体构想 → 用户审核/修改并确认 → 生成完整章节规划 → 选择第 1 章 → 生成完整章节 → 人工审核 → 定向修改 → AI 质检 → 用户确认正式保存 → 下一章。

成功标准：每一章都能追溯到总体构想和章节规划，生成失败不丢失已有部分，正式章节必须经用户确认。

### 场景 C：已有草稿继续创作

进入项目 → 从章节库或草稿打开上一章 → 创建版本快照 → 选择继续写作或修改 → 查看候选结果 → 接受/拒绝修改 → 保存新版本或正式章节 → 导出。

成功标准：原稿始终可恢复，用户能明确区分当前编辑稿、AI 候选稿和正式版本。

### 场景 D：分析作品

创建分析项目 → 导入合法语料 → 扫描和校验章节 → 启动风格分析 → 构建知识库 → 浏览人物、地点、主题和情节 → 将分析结果用于后续规划或导出。

成功标准：分析流程可以独立完成，不强迫用户生成续写内容。

## 5. 功能审计

### 5.1 必须保留

- 项目内语料导入、扫描、章节列表、详情和统计
- 风格分析与风格档案
- 人物、地点/世界、主题、情节和风格知识库
- 自动生成、修订、保存和确认总体构想
- 完整章节规划与单章计划维护
- 完整章节生成、续写和重新生成
- 局部修改、整章重写和人工编辑
- 规则完整性检查、AI 深度质检和 AI 修复建议
- 临时生成记录、草稿、正式章节和版本
- Markdown / TXT 导出
- 长任务进度、取消、部分成功、失败和保留部分结果
- 本地文件持久化、配置状态和隐私边界

### 5.2 应合并

| 当前能力 | 推荐产品能力 | 原因 |
| --- | --- | --- |
| 同步生成、流式生成、分段完整生成 | “生成完整章节”一个主操作 | 用户不需要选择传输协议；系统根据任务自动采用可靠方式 |
| 迭代修改、局部编辑、整章重写、AI 修复 | “修改章节”流程中的修改类型 | 都是对当前候选稿生成新候选版本 |
| 生成结果、草稿、临时生成记录 | “候选稿/临时记录” | 当前概念重叠，用户难判断保存层级 |
| 旧分析任务状态、统一任务、章节生成任务 | 统一“任务中心” | 用户只需要一种进度和恢复模型 |
| 总览统计与项目状态 | 项目概览 | 统计必须绑定当前项目并给出下一步 |
| 风格分析入口与知识库构建入口 | 分析中心 | 同属“从语料得到可复用知识”的流程 |

### 5.3 应重新命名

| 当前名称 | 推荐名称 | 说明 |
| --- | --- | --- |
| 系统总览 / 总览 | 首页（全局）或项目概览（项目内） | 区分跨项目入口和单项目状态 |
| 语料管理 | 语料库 | 核心是浏览、导入和分析来源，不是技术管理页 |
| 风格分析 | 分析中心 | 容纳风格、知识库和分析状态 |
| 续写工作台 | 创作 | 同时支持原创、续写和已有草稿 |
| 一键构想下一部 | 总体构想 | 去除特定续写语境和夸张“一键”承诺 |
| 临时生成记录 | 候选稿 | 强调尚未正式确认 |
| 正式章节库 | 章节库 | 默认展示正式章节，并提供版本与导出 |
| 配置状态 | 连接与环境诊断 | 面向用户说明是否可用及修复方式 |
| 任务日志 | 任务详情 | 普通视图展示阶段和可操作建议，技术日志折叠 |

### 5.4 应调整入口

- 知识库从生成页面移到“分析中心”，并在创作时作为只读上下文入口。
- 总体构想、章节规划和章节写作放在“创作”下形成连续步骤。
- AI 质检从独立工具变为当前章节工作流中的审核阶段，同时保留从章节详情重新质检的入口。
- 版本放在章节编辑器和章节详情中，不做一级导航。
- 导出放在项目、章节库和章节详情的上下文操作中，不做一级导航。
- 配置状态进入“设置 → 模型与连接”，首页只显示可理解的状态提示。
- 任务列表作为全局状态入口或通知面板，不占主导航。

### 5.5 应移动到高级设置

- API Base URL、Provider 兼容模式、端口、CORS
- token、Prompt 字符上限、超时、分段目标长度
- 数据根目录和具体项目路径
- 原始模型响应、解析警告和详细任务日志
- 原始 JSON 查看/下载
- 导入报告的技术明细和编码识别结果
- 缓存清理、重新解析、强制重建索引

### 5.6 应暂时隐藏

- 直接暴露同步/流式/分段生成方式的入口
- 原始总体构想重新解析入口
- 按 generation ID、task ID、file path 操作的入口
- 尚未形成稳定产品语义的旧草稿与生成结果重复入口
- 对普通用户无恢复价值的底层模型调用日志

隐藏表示不从主流程暴露，后端兼容接口在 Phase 2/后续阶段仍保留，不能直接删除。

### 5.7 可以删除的重复功能

Phase 1 不删除代码。Phase 2 以后，在兼容层、调用审计和测试完备后，可逐步废弃：

- 被统一任务 API 完全覆盖的旧任务查询接口
- 被“生成完整章节”覆盖的重复生成入口
- 被统一候选稿模型覆盖的旧 generation result / draft / temp 重复操作
- 仅展示同一数据、无独立用户目标的重复列表入口

删除前必须有 API 使用清单、弃用标记、迁移测试和至少一个兼容版本周期。

### 5.8 后续版本再做

- 桌面安装包和自动更新
- 云同步、多人协作和账号体系
- 插件/模型 Provider 扩展市场
- 跨项目共享语料和知识库模板
- 可视化人物关系、时间线和伏笔追踪
- EPUB / DOCX 等更多导出格式
- 全文语义检索和跨章节引用定位
- 移动端完整编辑体验

### 5.9 新旧功能决策总表

| 当前功能 | 决策 | 推荐名称/入口 | Phase 1 结论 |
| --- | --- | --- | --- |
| 总览 | 拆分并重命名 | 全局“首页” + 项目“项目概览” | 保留统计，但必须转化为项目状态和下一步 |
| 语料管理 | 保留并重命名 | 项目/语料库 | 去除固定文件数、作品名和路径文案 |
| 风格分析 | 保留并调整入口 | 项目/分析/风格 | 与知识库共享分析范围和任务状态 |
| 知识库 | 保留并调整入口 | 项目/分析/知识库 | 从生成页移出，创作时只作为上下文引用 |
| 自动构想 | 保留并重命名 | 创作/总体构想 | 同时支持原创和续写，不再限定“下一部” |
| 章节规划 | 保留 | 创作/章节规划 | 与总体构想和章节工作区形成顺序流程 |
| 章节生成 | 合并重复入口 | 章节工作区/生成完整章节 | 同步、流式和分段方式由系统选择 |
| 修改润色 | 合并并重命名 | 章节工作区/修改章节 | 局部修改、整章重写、AI 修复均产生候选版本 |
| AI 质检 | 保留并调整入口 | 章节工作区/AI 质检 | 作为正式保存前的推荐审核阶段 |
| 临时生成记录 | 合并并重命名 | 章节库/候选稿 | 与旧草稿、生成结果统一产品语义 |
| 正式章节库 | 保留并重命名 | 章节库/正式章节 | 与候选稿视觉和状态严格分离 |
| 版本 | 保留并调整入口 | 章节编辑/章节详情内 | 不设置一级导航，修改和覆盖前自动建快照 |
| 导出 | 保留并调整入口 | 章节、章节库、项目上下文操作 | 不设置一级导航；导出不改变正文状态 |
| 配置状态 | 保留并重命名 | 设置/模型与连接 | 普通视图只显示可用性和修复动作 |
| 任务日志 | 保留并分层 | 全局任务入口 + 任务详情 | 默认显示阶段与恢复，技术日志放高级详情 |

## 6. MVP 定义

### 6.1 MVP 必须包含

1. 创建、打开、切换和归档多个本地项目。
2. 项目级模型配置继承与连接检查。
3. 项目级语料导入、扫描、章节浏览和只读保护。
4. 风格分析与知识库构建。
5. 总体构想的生成、编辑和确认。
6. 完整章节规划的生成、编辑和状态管理。
7. 完整章节生成、候选稿保存、人工编辑和修改。
8. 规则检查、AI 质检和用户确认。
9. 正式章节、版本和 Markdown/TXT 导出。
10. 可恢复长任务、清晰错误、自动保存和数据位置说明。
11. 对现有 `longzu` / `longzu6` 数据的兼容读取与安全迁移。

### 6.2 MVP 可以推迟

- 跨项目资源共享
- 自动备份调度（MVP 先提供手动备份与迁移前备份）
- 复杂关系图和时间线
- 多人协作、云端和移动端
- 多种富文档导出
- 自定义工作流和插件系统

## 7. 产品成功标准

- 新用户首次启动后 5 分钟内知道如何创建项目和配置模型。
- 用户在任意项目页都能看到当前项目和回到项目概览。
- 生成、修改、质检和正式保存有明确顺序和状态。
- 刷新或重启后，项目、草稿、正式章节和可恢复任务仍可定位。
- 用户不会因为一次失败的模型请求丢失输入或已有候选稿。
- 公开仓库不包含真实密钥、版权语料或用户生成正文。
- 新用户可按 README 从零安装、启动并使用原创示例完成主流程。

## 8. 进入 Phase 2 前必须确认的决策

以下决策未确认前，不应开始多项目数据模型和迁移实现：

| 决策 | 推荐默认方案 | 主要取舍/待确认点 |
| --- | --- | --- |
| 项目标识 | 新项目使用不可变 UUID；显示名称独立；旧 `longzu6` 作为兼容 legacy ID/alias | UUID 最稳健但迁移映射更复杂；受限 slug 更易读但易与重命名混淆 |
| 默认数据根目录 | 新项目使用独立用户数据根目录并允许自定义；旧目录原地兼容 | 是否在 MVP 就改变新项目默认位置，需要用户确认 |
| 旧项目策略 | 默认原地只读识别 + 明确的可选复制迁移；不自动移动或删除 | 原地兼容实现较复杂，但数据风险最低 |
| 模型配置层级 | 全局默认 + 项目覆盖 | 兼顾普通用户与高级用户；项目 manifest 不保存真实密钥 |
| 语料归属 | 每个项目保存自己的语料引用；允许引用外部只读目录，不在项目间隐式共享分析缓存 | 同源语料可能重复分析，但隔离更清晰 |
| 原创项目 | 允许完全跳过语料与分析，从手工设定或总体构想开始 | 生成页面需明确“上下文较少”，不能硬阻塞 |
| 候选稿模型 | 统一 generation result、draft、temp generation 为候选稿，并保留旧格式适配器 | 需要兼容映射和迁移测试，不能直接删除旧文件 |
| 正式保存 | 保存前自动创建版本；首次正式保存确认一次，覆盖正式章节二次确认 | 增加一步确认，但显著降低正文丢失风险 |
| 删除策略 | MVP 使用软删除/回收站；永久删除放高级操作 | 需要回收目录和清理策略 |
| 分析前置 | 风格分析与知识库默认推荐但可跳过 | 允许原创和快速写作，同时显示一致性影响 |
| API 项目上下文 | 使用 `/api/projects/{project_id}/...` 显式路径 | 路由较长，但可测试、可审计且不依赖隐式当前项目 |
| 任务演进 | Phase 2 先加入 `project_id`、资源关联和落盘恢复所需字段；完整持久化可单独分阶段 | 避免把数据迁移与任务引擎重写混在一次改动中 |
| 项目备份 | ZIP 快照 + manifest；默认可选择是否包含原始语料，迁移前必须备份项目派生数据 | 含语料备份体积大且涉及版权责任 |
| 导航术语 | 确认“首页、项目、项目概览、语料库、分析、创作、章节库、设置” | 若术语调整，应在 Phase 2 API/数据命名之前完成 |

这些决策的推荐方案和页面影响详见 [02-information-architecture.md](02-information-architecture.md)、[user-flows.md](user-flows.md) 与 [page-specifications.md](page-specifications.md)。
